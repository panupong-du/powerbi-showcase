# 📅 การสร้าง Date Table (Calendar Table) ใน Power BI: DAX vs Power Query

ในการสัมภาษณ์ตำแหน่ง Power BI Developer หรือ Data Analyst หัวข้อนี้มักจะถูกถามเพื่อทดสอบความเข้าใจเรื่อง **Data Modeling** และ **Performance Optimization**

การสร้างตารางวันที่ (Date Table) หรือที่ใน Qlik เรียกว่า Master Calendar สามารถทำได้ 2 วิธีหลักๆ ดังนี้:

---

## 1. สร้างด้วย DAX (`CALENDAR()` หรือ `CALENDARAUTO()`)
วิธีนี้คือการพิมพ์สูตรเพื่อสร้างตารางหลังจากที่โหลดข้อมูลเข้ามาใน Data Model แล้ว

### ✅ ข้อดี:
*   **รวดเร็วและง่ายมาก:** พิมพ์แค่ไม่กี่ตัวอักษรก็ได้ตารางเริ่มต้นแล้ว
*   **Dynamic อัตโนมัติ:** `CALENDARAUTO()` จะขยายช่วงเวลาวันที่ตามข้อมูลใน Data Model ให้อัตโนมัติ

### ❌ ข้อเสีย:
*   **Reuse ยุ่งยาก:** นำไปใช้กับไฟล์อื่นลำบาก ต้องมานั่งพิมพ์สูตร (Year, Month, Quarter) ใหม่ทุกครั้ง
*   **กินหน่วยความจำ:** ตารางที่สร้างจาก DAX จะกินพื้นที่ใน RAM (Memory Engine)
*   **เพิ่มเงื่อนไขธุรกิจยาก:** ถ้าต้องใส่วันหยุดประจำปีของบริษัท หรือปีงบประมาณ (Fiscal Year) ที่มีความซับซ้อน จะต้องเขียนสูตร `IF/SWITCH` ซ้อนกันเยอะมากจนอ่านยาก

---

## 2. สร้างด้วย Power Query (M Code)
วิธีนี้คือการสร้างตารางในกระบวนการเตรียมข้อมูล (ETL) ก่อนโหลดเข้าโมเดล

### ✅ ข้อดี (⭐️ Best Practice):
*   **นำไปใช้ซ้ำได้ทันที (Highly Reusable):** สามารถ Copy โค้ด M ไปใช้ไฟล์อื่นได้ หรือสร้างเป็น **Dataflow** เพื่อเป็น Master Date Table ชุดเดียวให้ดึงไปใช้ทั้งบริษัทได้เลย
*   **เชื่อมโยงแหล่งข้อมูลอื่นง่าย:** สามารถ Join (Merge) กับไฟล์ Excel หรือ Database ที่เก็บรายชื่อ "วันหยุดประจำปีบริษัท" ได้ง่ายๆ
*   **Performance ดีกว่า:** ปฏิบัติตามหลักการ "Push transformations as close to the source as possible" ซึ่งเป็นการผลักภาระไปให้กระบวนการโหลดข้อมูล แทนที่จะมากินแรง DAX Engine

### ❌ ข้อเสีย:
*   ต้องมีความเข้าใจภาษา M (M Code) เบื้องต้น
*   เพิ่มเวลาในขั้นตอน Data Refresh เล็กน้อย

---

## 💡 แนวทางการตอบสัมภาษณ์ (Interview Tips)
หากกรรมการถามว่า "คุณถนัดสร้าง Date Table ด้วยวิธีไหน" คำตอบที่จะแสดงความโปรขั้นสุดคือ:

> *"ถ้าเป็นโปรเจกต์สั้นๆ หรือต้องการแค่ทำ Mockup ผมจะใช้ **DAX** เพราะมันรวดเร็วครับ แต่ถ้าเป็นระดับ Enterprise หรือโปรเจกต์ที่ต้องนำไปใช้จริง ผมจะสร้างใน **Power Query** เสมอ เพราะมันจัดการและแก้ไขง่ายกว่า สามารถนำไป Join กับปฏิทินวันหยุดบริษัทได้ และที่สำคัญคือผมสามารถเซฟ Script เป็น Dataflow เพื่อให้ Report อื่นๆ ดึงไปใช้ซ้ำ (Reuse) ได้เลย ซึ่งเป็น Best Practice ในการลดภาระของโมเดลครับ"*
